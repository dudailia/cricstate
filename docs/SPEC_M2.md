# SPEC_M2 — Module 2: Outcome Model & Evaluation Harness

**Version:** 1.0 · **Frozen:** 2026-07-03 · **Document reconstructed:** 2026-07-04 · **Status:** Frozen (release)
**Scope:** Per-ball and per-match outcome models over the Module 1 state tables, and the evaluation harness — metrics, calibration, decision rule, leakage canaries, and single-test-touch discipline — that judges them.

This module contains all of the project's machine learning. Its value is not any single model but the **harness**: a pre-committed way of deciding whether a model earns its place, applied to the test split exactly once.

## 1. Responsibilities

The module owns exactly six things:

1. The two prediction **tasks** over Module 1's `deliveries` table: **T1 — per-ball outcome** (the frozen 11-class alphabet) and **T2 — win probability** (binary, resolved from the match result).
2. A **baseline ladder** B0→B3 of increasing capacity, each a pure function of within-match state (§5).
3. The **metric contract** — negative log-likelihood (primary), multiclass Brier, and expected calibration error (§3).
4. **Calibration** of every shipped predictor, fit on validation only (§5).
5. The **decision rule** (§6) and the **single test touch** that applies it.
6. **Leakage canaries** and freeze enforcement (§7) that make the numbers trustworthy rather than merely asserted.

**Non-responsibilities:** no state reconstruction (that is Module 1), no odds ingestion, no vision, no online serving. The module reads Parquet and writes a leaderboard.

## 2. Inputs

- The Module 1 `deliveries`, `matches`, and `players` Parquet tables — one row per ball with full pre-ball state as flat columns (SPEC_M1 §3). Module 2 begins as a `read_parquet` call.
- **Corpus pin.** Snapshot tag `v1.2`; corpus SHA-256 and label SHA-256 are pinned in `src/evalkit/freeze.py` (`PINNED_CORPUS_HASH`, `PINNED_LABELS_HASH`) and enforced as a red-build test. Drift in either hash fails the build (§7).
- **Temporal split**, derived from `start_date` and baked into the tables (SPEC_M1 §10), never random:
  - train — through 2024-11-02
  - val — 2024-11-03 → 2025-08-29
  - test — 2025-08-30 → 2026-06-28
  All hyperparameter tuning and calibration use train/val only. The test split is read **once**, at release, under the rule in §6.
- **Two tasks, two scales.** T1 is an 11-class outcome distribution (test NLL ≈ 1.6 nats); T2 is a binary win-probability (test NLL ≈ 0.5). The information-decomposition thesis (state vs identity vs conditions) is measured on **T1**; T2 is the headline product surface. The frozen K=11 alphabet and its order live in `src/evalkit/models/base.py` (`CLASSES`).
- **Thin cells.** Any (task, fmt) cell with < 300 labeled validation matches (currently ODI) is flagged THIN and its leaderboard calibration switches per Amendment #1 (§5, changelog).

## 3. Outputs

The single release artifact is `docs/LEADERBOARD.md`, regenerated byte-identically by `evalkit run-all`, backed by `results/summary.json` (the machine-readable single source of truth from which all figures and tables are derived).

**Metric contract** (`src/evalkit/metrics.py`):

- **NLL (primary).** Mean per-ball negative log-likelihood of the true class. All model comparisons and the decision rule are stated in NLL.
- **Multiclass Brier.** Reported alongside NLL as a proper-scoring cross-check.
- **ECE.** Per-class expected calibration error over 20 equal-mass bins, reported per class and as a mean; plus a max calibration error.
- **Skill vs B0.** Relative NLL improvement over the marginal baseline, per cell.

Every headline delta carries a **95 % confidence interval** from the match-level paired bootstrap (§6): B = 10,000 resamples, seed 90210, resampling **matches**, never balls (balls within a match are dependent; ball-level resampling would be spuriously tight and is forbidden — see FAQ).

## 4. Features

The B2/B3 feature set is a **whitelist of within-match state features**, all pure functions of the pre-ball state computed at read time (`src/evalkit/features.py`; SPEC_M1 §6 supplies the state). The frozen fingerprints record `n_features = 27` for the T1/T20 B3 cell. Leakage rules are structural, not procedural:

- No feature may read any `outcome.*` column. This is enforced by a canary that proves the output is byte-identical with the outcome column present vs absent (§7, `tests/test_features.py`).
- Any cross-match aggregate (player or venue) must be rolling-as-of-date; none may look ahead. In Module 2 as released, only within-match state features are used; player identity enters solely as the isolated Branch A increment (§6).

## 5. Model choices — the baseline ladder

Four models of increasing capacity, each behind the `Predictor` registry (§8). All tuning is validation-only; the tuned knob is named per cell on the leaderboard.

| model | form | val-tuned knob |
|---|---|---|
| **B0_marginal** | class marginals (Laplace-floored) | — |
| **B1_table** | binned state lookup table | shrinkage τ |
| **B2_logistic** | multinomial logistic over §4 features | regularization C |
| **B3_gbm** | histogram gradient-boosted trees over §4 features | early-stop iterations |

**Calibration** (`src/evalkit/calibrate.py`), fit on val only: temperature scaling for T1; isotonic for T2 t20; Platt for T2 thin cells (Amendment #1). **Monotonicity** (`src/evalkit/monotonicity.py`): the T2 B1 table is made monotone by construction via weighted alternating PAVA over each (innings, phase) lattice (Amendment #3); if PAVA measurably worsens val NLL beyond noise the rule is to STOP and report, not to tune around it.

B3 is the incumbent state model. The enrichment experiments (Branch A identity, Branch C conditions) are **increments on top of the frozen B3**, each with one tunable, so that identity and conditions are isolated rather than entangled.

## 6. Decision rule — the pre-registered part, stated exactly

This is the load-bearing section. It has two distinct gates. They are **different rules** and were **frozen at different times**; earlier docs conflated them under one name. The honest, git-checkable account:

### 6a. What was frozen *before any test split was read* (P3, commit `c584d62`, 2026-07-03)

Committed at the P3 gate, before the test set was ever touched — verifiable via `git show c584d62`:

- The complete baseline ladder B0–B3 and their **val-only** tuning protocol (§5).
- The metric contract: NLL primary, Brier, per-class ECE (§3).
- The calibration policy: temperature / isotonic / Platt, **fit on val only** (`policy.py`, `calibrate.py`).
- The **leakage canaries**, including the **ladder-inversion STOP rule**: if a higher rung scores worse than a lower rung by more than `EPSILON_B3_VS_B2 = 0.005` NLL, STOP and report — never tune to fix (`src/evalkit/canaries.py`).
- **Freeze enforcement**: corpus and label hash pins as red-build tests (`src/evalkit/freeze.py`).
- The **single-test-touch discipline itself**: the P3 report (`docs/P3_VAL_REPORT.md`) is explicitly validation-only, test untouched.

In short: the **measurement apparatus and the discipline** were pre-committed. The evaluation was built to be trusted before any test number existed.

### 6b. The challenger "beats-the-bar" gate (ladder release)

Applied when a challenger is compared to the best incumbent baseline. Three clauses, all required (`src/evalkit/run.py`):

1. ΔNLL 95 % CI excludes zero (improving) on **both** val and test.
2. Relative NLL improvement ≥ **0.5 %**.
3. No calibration regression: ΔECE ≤ **0.005**.

The gate's clause thresholds are defined in `run.py`, which entered at the P4 leaderboard release (commit `08ed836`) — the **same commit as the one-time test evaluation**, not before it. What preceded the test read was the apparatus of §6a (ladder, metrics, canaries, single-touch discipline), not these specific thresholds. The gate is demonstrated live on the leaderboard: on T2/ODI (a thin cell) B3 **fails** clause 1 and is reported as "did not beat the bar" — the framework refuses a close call rather than certifying it.

### 6c. The enrichment verdict bands (Branch A identity, Branch C conditions)

Applied to an isolated increment (e.g. player identity) on top of frozen B3, gated first by a shuffled-identity canary:

- **JUSTIFIES_TOWER** — CI excludes zero (improving) **and** relative NLL improvement ≥ **1.0 %**.
- **AMBIGUOUS** — improving **and** ≥ **0.3 %** but < 1.0 %.
- **KILL_BRANCH** — otherwise.
- **VOID** — if the canary fails.

Thresholds: `REL_JUSTIFY = 1.0 %`, `REL_AMBIG_LO = 0.3 %` (`experiments/branch_a/run.py`). **Provenance, stated plainly:** these bands were fixed as the Branch A protocol and encoded as constants; the code records that *"no separate Branch A spec file exists; the milestone prompt is the spec"* (`experiments/branch_a/run.py`). The bands were committed in the **same commit as the identity test touch** (`0dc5425`), not before it. The credibility of the identity verdict therefore rests on the **single documented test touch** (all models evaluated together, once), the **shuffled-identity canary** gating the verdict, and **byte-identical reproducibility** — **not** on the thresholds having been committed to git before the test was seen. The measured result (identity real but immaterial, AMBIGUOUS at the band floor) is reported either way; the rule does not flatter it.

### 6d. The one prohibition

Renegotiating any of these rules after seeing a result is forbidden. Amendments (below) are additive, dated, and recorded at the gate that motivated them; none edits a decision threshold after a result.

## 7. Leakage & freeze enforcement

- **Outcome-column canary** — the T1 pipeline produces byte-identical output whether or not the outcome column is present, proving no feature reads it (`tests/test_features.py`).
- **Ladder-inversion canary** — §6a STOP rule at 0.005 NLL (`src/evalkit/canaries.py`).
- **Shuffled-target / shuffled-identity canaries** — a model fit on permuted targets (or permuted player IDs) must collapse to the state baseline; a live signal there would indicate a leak. The data-driven canaries run in the validation report script and the Branch A single touch; the structural (corpus-free) canaries run in CI.
- **Hash pins** — corpus and label SHA-256 in `freeze.py`; drift fails the build. Golden Parquet schemas under `tests/golden_schemas/` are the contract; a schema change requires an explicit golden update in the same commit.

## 8. APIs between modules

The `Predictor` protocol is the model contract (`src/evalkit/models/base.py`):

```python
def fit(self, train: pl.DataFrame, val: pl.DataFrame) -> None: ...
def predict_proba(self, df: pl.DataFrame) -> np.ndarray:  # [n, K] for T1, [n] for T2
```

Every baseline and every challenger implements it and registers in the model registry, so `evalkit run-all` (§10) discovers and evaluates them uniformly. Reserved forward-looking stubs (`OddsProvider`, as-of feature store) are declared but unused in v1. The CLI entry point is `evalkit = evalkit.cli:main`.

## 9. Computational complexity & runtime budget

- Fits are deterministic (seed 1337 end-to-end) and cached; the cache is fingerprinted by corpus hash, labels hash, model version, seed, and feature count, so a stale cache refits rather than loading silently (`src/evalkit/cache.py`).
- Budget (Amendment #4): cold fit < 60 min; cached eval-only `run-all` < 5 min. Single process, single machine — no distributed anything.
- Storage: the fitted-model + calibration cache under `artifacts/`; the release numbers under `results/`.

## 10. Training & inference interface

- `evalkit run-all` fits (or loads from cache) every registered model, calibrates on val, evaluates, and regenerates `docs/LEADERBOARD.md` + `results/summary.json`. Two consecutive invocations produce **byte-identical** output (no wall-clock content).
- `evalkit run-all --cold` ignores the cache and refits from scratch, for full-provenance reproduction.
- The as-of feature-store seam is declared for future rolling covariates; nothing in v1 conditions on look-ahead aggregates.

## 11. Failure modes (each surfaced, not hidden)

1. **Thin cell** (< 300 labeled val matches) — flagged THIN; calibration switches to Platt (Amendment #1); wide CIs reported honestly (T2/ODI).
2. **Ladder inversion** — a higher rung worse than a lower rung beyond 0.005 → STOP and report (§7).
3. **Monotonicity violation** — pre-PAVA B1 T2 violations are reported at the gate; post-PAVA monotonicity holds by construction (Amendment #2, #3).
4. **Challenger fails the bar** — reported as "did not beat the bar," not suppressed (T2/ODI).
5. **Enrichment below materiality** — reported as AMBIGUOUS or KILL; the increment ships only if the rule says so.

## 12. Testing & known-answer checks

- **§12.1 — Known-answer metric tests.** Each metric (NLL, Brier, ECE) is tested against hand-computed values on small fixtures, plus supporting properties (e.g. ECE of a synthetically calibrated set stays below tolerance) (`tests/test_metrics.py`).
- **Property tests** over the models and calibration: logit-addition math on null/unseen rows, unshrunk-empirical equivalence, unimodality of the val tuning curve (`tests/`, `experiments/branch_a`).
- **Golden schema tests** freeze the Parquet column contract; a schema change requires an explicit golden update in the same commit (`tests/golden_schemas/`, §7).
- **Leakage canaries** (§7) run as tests (structural, corpus-free) and in the report scripts (data-driven).

## 13. Definition of Done (all required)

- `evalkit run-all` regenerates `docs/LEADERBOARD.md` byte-identically across two runs.
- Corpus and label hash pins pass; golden schemas unchanged.
- All canaries pass (or a STOP is reported).
- Every shipped predictor is calibrated on val only; the test split is read exactly once.
- The decision rule (§6) is applied as written, and its verdict — win or refusal — is what the leaderboard reports.

When green, Module 3 (the open challenger arena) begins: any model implementing the `Predictor` protocol (§8), tuned on val only, faces the frozen §6b bar with one test touch — and the open conditions question and richer-data extensions (ball-tracking, venue covariates) are fair game, scoped honestly as changing the question rather than renegotiating the rule.

- **Amendment #3 (P4 gate, 2026-07-03) — B1 monotone by construction:**
  post-shrinkage, B1 (T2) applies weighted alternating PAVA over each
  (innings, phase) lattice: non-decreasing along the rate band (CRR/RRR
  ascending) and along the wickets axis non-increasing for innings 1 /
  non-decreasing for innings 2. Fit on TRAIN quantities only. The
  monotonicity property tests must pass by construction; if PAVA measurably
  worsens B1 val NLL beyond noise, STOP and report.
- **Amendment #4 (P4 gate, 2026-07-03) — artifact cache & runtime budget:**
  P4 caches fitted model + calibration artifacts; run-all re-evaluates from

  cached eval-only run-all < 5 min. Fits are deterministic and the cache is
  fingerprinted (corpus hash, labels hash, model version, seed) so validity
  is checkable.
